# 最终发布审批手册

## 一、目标

最终发布审批不是再运行一次本地测试，而是确认同一个 Git 提交和不可变候选镜像已经同时通过生产规格验收与真实浏览器验收。审批链不接受手工填写的指标、本地构造的 JSON 或普通镜像标签。

## 二、触发顺序

```text
发布候选全链路验收成功
→ 真实预发布前后端全链路验收自动运行
→ 最终发布审批工作流自动校验证据
→ production-approval Environment 等待人工批准
→ 固化最终审批记录
```

上游任一工作流失败、取消、来自非 `main` 分支或缺少产物时，后续步骤不会进入人工审批。

## 三、审批环境

在 GitHub 仓库中创建 `production-approval` Environment，并配置：

- 至少一名必要审核人。
- 禁止发起人自审。
- 只允许受保护的默认分支部署。
- 环境秘密 `FINAL_RELEASE_APPROVAL_GUARD`。

守卫秘密应使用随机值，只用于证明该 Environment 已被维护者显式初始化，不参与应用认证。不要把值写入仓库、脚本参数、Actions 日志或本地 `.env`。若环境被意外自动创建但没有配置守卫，审批作业会立即失败。

## 四、证据校验

`new-final-release-approval-packet.ps1` 会验证：

- staging E2E 证据状态成功且属于 `staging`。
- E2E、冻结候选和待审批提交完全一致。
- E2E 证据包含有效的上游发布候选验收运行编号。
- 候选镜像使用 `image@sha256` 不可变地址。
- 冻结清单包含预检、秘密轮换、灰度、容量、故障注入和灾备六类证据。
- 原始 E2E 证据和冻结清单的 SHA-256 被写入审批候选包。

人工批准后生成 `final-release-approval.json`，并与审批候选包共同保留 365 天。GitHub Environment 的部署历史负责记录实际审核流程，JSON 不伪造审核人身份。

## 五、本地边界测试

```powershell
./scripts/tests/final-release-approval-tests.ps1
```

该测试只使用临时伪造数据验证拒绝规则，证明跨提交证据和缺少灾备证据时无法生成审批包。测试成功不代表真实发布已获批准。

## 六、发布边界

最终审批记录生成后，才允许创建正式版本标签、GitHub Release 或执行生产流量切换。仓库当前不会自动创建标签或修改生产环境，避免审批完成与实际发布时间被错误绑定；正式发布动作应继续保持显式、可回滚并由维护者确认。
